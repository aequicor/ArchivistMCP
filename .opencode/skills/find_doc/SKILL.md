# FIND_DOC

Скилл для поиска и автоматического пополнения базы документации ArchivistMCP.

## Назначение

Скилл принимает на вход описание того, какую информацию нужно получить: произвольный текст, названия библиотек, технологий, фреймворков, концепций и т.п. На выходе пользователь получает релевантную документацию — либо из локальной базы, либо созданную на основе интернет-поиска и сохранённую в базу для последующих запросов.

## Входные данные

Скилл принимает запрос в свободной форме. Примеры:

- `"Ktor routing"` — документация по конкретной технологии
- `"как настроить CI/CD в GitHub Actions"` — вопрос в естественной форме
- `"kotlinx.serialization, kotlinx.coroutines"` — список библиотек
- `"OAuth 2.0 flow, JWT tokens"` — набор концепций

## Алгоритм работы

```
┌─────────────────────────────────────────────┐
│   Входной запрос (текст / библиотеки)      │
└────────────────────┬────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────┐
│   Шаг 1: semantic_search(query)             │
└────────────────────┬────────────────────────┘
                     │
              ┌──────┴──────┐
              │             │
         найдено          не найдено
              │             │
              ▼             ▼
  ┌─────────────────┐   ┌──────────────────────┐
  │ Вернуть         │   │ Шаг 2: WebSearch     │
  │ найденные       │   │         + WebFetch   │
  │ документы       │   └──────────┬───────────┘
  └─────────────────┘              │
                                   ▼
                        ┌──────────────────────┐
                        │ Шаг 3: Сформировать  │
                        │ документы по         │
                        │ docs/doc-template.md │
                        └──────────┬───────────┘
                                   │
                                   ▼
                        ┌──────────────────────┐
                        │ Шаг 4: add_document  │
                        │ для каждого          │
                        │ созданного документа │
                        └──────────────────────┘
```

### Шаг 1. Поиск в локальной базе

Вызвать MCP-инструмент `semantic_search`:

```json
{
  "query": "<входной запрос>"
}
```

**Если** результат содержит хотя бы один документ — вернуть его пользователю и завершить работу. Документы уже проиндексированы, повторная работа не требуется.

**Если** результатов нет (пустой массив `results`) — перейти к шагу 2.

### Шаг 2. Поиск в интернете

Когда в локальной базе ничего не найдено, использовать инструменты интернет-поиска:

- `WebSearch` — для получения списка релевантных источников
- `WebFetch` — для загрузки содержимого конкретных страниц (официальная документация, GitHub README, статьи)

Приоритет источников:

1. Официальная документация (kotlinlang.org, ktor.io, docs.github.com и т.п.)
2. Официальные репозитории на GitHub (README, wiki)
3. Авторитетные технические ресурсы (Baeldung, Medium-статьи признанных авторов)
4. Stack Overflow — только как дополнение

Если запрос содержит несколько сущностей (например, список библиотек), выполнить поиск по каждой отдельно.

### Шаг 3. Создание документов по шаблону

Шаблон находится в [docs/doc-template.md](docs/doc-template.md):

```markdown
# {name}

**Keywords:** {keywords}

## Abstract

{abstract}

## Documentation

{documentation}
```

**Правила заполнения:**

| Плейсхолдер | Содержимое |
|-------------|------------|
| `{name}` | Название библиотеки/технологии/концепции |
| `{keywords}` | Ключевые слова через запятую (для улучшения семантического поиска) |
| `{abstract}` | Короткое описание (2–4 предложения) — назначение, область применения |
| `{documentation}` | Основное содержимое: API, примеры кода, ссылки на источники |

**Правила именования файлов:**

- Использовать `kebab-case`
- Расширение `.md`
- Для библиотек — полное имя пакета (`kotlinx-serialization.md`)
- Для концепций — краткое описание (`oauth2-flow.md`)
- Для многокомпонентных запросов — отдельный файл на каждую сущность

### Шаг 4. Индексация документов

Для каждого сформированного документа вызвать MCP-инструмент `add_document`:

```json
{
  "filename": "<имя-файла>.md",
  "content": "<полное содержимое документа>"
}
```

После успешного вызова (`{"status": "ok", ...}`) документ становится доступен для последующих запросов через `semantic_search` и `smart_search`.

В конце работы вернуть пользователю созданные документы (или сводку по ним).

## Используемые инструменты

| Инструмент | Назначение | Шаг |
|------------|------------|-----|
| `semantic_search` | Поиск по локальной базе | 1 |
| `WebSearch` | Поиск страниц в интернете | 2 |
| `WebFetch` | Загрузка содержимого страниц | 2 |
| `add_document` | Индексация нового документа | 4 |

## Примеры

### Пример 1. Документ уже есть в базе

**Запрос:** `"Ktor routing"`

1. `semantic_search({"query": "Ktor routing"})` → `[{"filename": "ktor-routing.md", "score": 0.91}]`
2. Вернуть `ktor-routing.md` пользователю.

### Пример 2. Документа нет, один источник

**Запрос:** `"kotlinx.serialization"`

1. `semantic_search({"query": "kotlinx.serialization"})` → `[]`
2. `WebSearch("kotlinx.serialization official documentation")` → ссылки на github.com/Kotlin/kotlinx.serialization и kotlinlang.org
3. `WebFetch(...)` → содержимое страниц
4. Сформировать документ:
   ```markdown
   # kotlinx.serialization

   **Keywords:** kotlin, serialization, json, protobuf, cbor, multiplatform

   ## Abstract

   Официальная библиотека сериализации для Kotlin. Поддерживает JSON, ProtoBuf, CBOR и др. форматы, работает в Kotlin Multiplatform.

   ## Documentation

   ...
   ```
5. `add_document({"filename": "kotlinx-serialization.md", "content": "..."})` → `{"status": "ok", ...}`

### Пример 3. Несколько библиотек

**Запрос:** `"kotlinx.coroutines, kotlinx.serialization, ktor-client"`

1. `semantic_search` — ничего не найдено.
2. Для каждой библиотеки отдельный `WebSearch` + `WebFetch`.
3. Создать три документа: `kotlinx-coroutines.md`, `kotlinx-serialization.md`, `ktor-client.md`.
4. Вызвать `add_document` трижды — по одному разу на каждый файл.

## Принципы

- **Fail-fast на шаге 1.** Если документация уже есть — не тратить ресурсы на интернет-поиск.
- **Один источник истины — официальная документация.** Материалы третьих сторон допустимы только как дополнение.
- **Атомарность документов.** Один файл — одна сущность (библиотека / концепция / фреймворк).
- **Keywords для будущих запросов.** Поле `Keywords` должно включать синонимы и смежные термины — это повышает полноту семантического поиска.
